Conceptual Ideas

Treat billing-period state as a data-layer contract, not a UI workflow. open -> closed -> finalized should be enforced by repository and database invariants so web, CLI, and future jobs all obey the same immutability rules.
Separate imported facts from review decisions. For finance-like workflows, rejected items should usually become dispositions on immutable source rows, not deletions.
Decide what a “statement” is in the data model. The code sometimes treats it as one PI artifact and sometimes as many project artifacts; that mismatch is why pdf_path turned into a semicolon string.
Make auditability a first-class feature with a coverage map. Every privileged mutation should answer the same questions: who, what record, before/after, why, and where it originated.
Tests

The tests are not fake. There are 239 test functions, and files like tests/test_ingest.py, tests/test_repository.py, and tests/test_journal_template.py do real work around parsing, flagging, repository behavior, and export rendering.
The gap is coverage shape, not effort. I found essentially no web-layer/auth/route integration coverage: no TestClient tests for login/session handling, review authorization, imports, statement delivery, or email resend.
The suite mostly proves happy-path business logic and helpers. It does not meaningfully defend the security boundaries or the core invariants that are currently drifting between docs, routes, and repository code.
I could not execute the suite here because pytest and runtime deps like sqlalchemy are not installed in this environment, so the test assessment is static rather than runtime-verified.
Assumptions

I assumed the Docker/Compose path is a real deployment path, not just local scaffolding; if so, the session-secret issue is urgent.
I assumed the security docs are meant as operational truth. If that’s right, the doc/code mismatches are themselves a risk because they create false assurance during incidents and audits.
